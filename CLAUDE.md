# AI-Notes 项目配置

## 项目简介

这是一个面向新手的 AI 工具使用笔记项目，记录 Claude Code 及其他 AI 工具的实用经验。

## 🔒 隐私保护规则（最重要！）

**核心原则**：AI-Notes 是公开分享的教程项目，**绝对不能暴露任何真实的个人信息**。

### 禁止暴露的信息

❌ **严禁在文档中包含以下真实信息**：
- 真实 IP 地址（如 192.168.50.x）
- 真实用户名（如 twinsenliang、mm 等）
- 真实主机名（如 RaspberryPi5、MagicMirror 等）
- 真实工作目录路径（如 /Users/twinsenliang/Sites）
- 个人项目名称和配置细节
- API 密钥、Token、密码等敏感信息

### 使用通用示例

✅ **文档中必须使用通用化的示例**：
- IP 地址：使用 `192.168.1.100`、`192.168.1.101` 等示例
- 用户名：使用 `user`、`pi` 等通用名称
- 主机名：使用 `my-server`、`device1`、`device2` 等
- 工作目录：使用 `~/Workspace`、`~/Projects` 等通用路径
- 项目名：使用 `MyProject`、`example-app` 等通用名称

### 编写规范

1. **新增文档时**：从一开始就使用通用示例
2. **修改文档时**：检查并替换所有真实信息
3. **提交前检查**：确保没有遗漏的隐私信息
4. **代码示例**：使用可复制但通用化的命令

### 示例对比

```markdown
❌ 错误示例：
./setup-mirrors.sh twinsenliang@192.168.50.129
主机名: RaspberryPi5
工作目录: /Users/twinsenliang/Sites

✅ 正确示例：
./setup-mirrors.sh user@192.168.1.100
主机名: my-server
工作目录: ~/Workspace
```

### 提交前检查清单

在提交任何文档修改前，必须确认：
- [ ] 没有真实 IP 地址
- [ ] 没有真实用户名
- [ ] 没有真实主机名
- [ ] 没有真实路径信息
- [ ] 没有 API 密钥等敏感信息

**这是保护个人隐私和安全的最重要规则！**

## 文案风格指南 ✍️

### 核心原则

**目标受众**：新手用户，可能是第一次接触 Claude Code

**风格要求**：
- 😊 **俏皮轻松**：像朋友聊天一样，别太正式
- 🎯 **简单易懂**：少用专业术语，多用通俗比喻
- 💡 **实用至上**：直接给答案，少讲大道理
- 🚀 **快速上手**：让读者看完就能动手操作

### 具体要求

#### 1. 语气
- ✅ 使用"你"而不是"您"
- ✅ 适当使用 emoji 让内容更生动
- ✅ 可以用"别慌"、"搞定"、"踩坑"等轻松词汇
- ❌ 避免过于学术化的表述

**示例**：
```
❌ 不推荐：根据官方文档的描述，您需要执行以下配置流程
✅ 推荐：别慌！跟着这几步操作就能搞定 👇
```

#### 2. 结构
- 使用清晰的标题层级
- 善用对比（❌ 错误示范 vs ✅ 正确做法）
- 重要内容用引用块或加粗突出
- 适当使用代码块展示命令

#### 3. 内容
- **先说结论**：别绕圈子，直接说重点
- **举例说明**：抽象概念一定要配实际例子
- **避坑指南**：新手容易踩的坑要特别标注
- **总结提炼**：长文章要有 TL;DR 或总结

#### 4. Emoji 使用规范
- 🎯 核心要点
- 💡 小贴士
- ⚠️ 注意事项
- ✅ 正确做法
- ❌ 错误做法
- 🚀 进阶技巧
- 💰 省钱/效率相关
- 🤔 思考问题
- 📚 参考资料
- ✍️ 作者说明

## 写作时的注意事项

### ✅ 好的例子
```markdown
### 省钱秘诀：话要少，一次问完！

❌ **啰嗦版**（浪费 token）
你能不能帮我看一下这个文件？

✅ **简洁版**（省钱）
分析这个文件
```

### ❌ 避免的例子
```markdown
### Token 优化策略的具体实施方案

根据 Claude API 的计费模型，我们可以通过减少输入 token 的数量来降低成本...
```

## 技术术语处理

遇到专业术语时：
1. **第一次出现**：用通俗语言解释（可用括号或→符号）
2. **后续出现**：可直接使用
3. **复杂概念**：用生活化比喻

**示例**：
```
Token（就是字符单位）消耗太快 → 钱包受不了

Task 工具（让 Claude 开小号去探索）→ 主号不计费
```

## Git 提交规范

### 提交确认流程

**重要**：当修改了本项目目录中的文件后，必须询问用户是否进行 git 提交。

工作流程：
1. 完成文件修改后，列出修改内容
2. 明确询问：「是否需要提交这些修改到 Git？」
3. 用户确认后再执行 git 操作
4. 不要主动提交，等待用户明确指示

### 提交信息规范

提交信息简洁明了：
```bash
# ✅ 好的提交
添加 token 节省技巧文档
更新 CLAUDE.md 文案风格说明
修复安装指南中的错误链接

# ❌ 不好的提交
update docs
fix
修改文件
```
